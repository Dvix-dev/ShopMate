=============================================================
 ShopMate · guia del emulador local (RTDB + Auth)
=============================================================
 OBJETIVO: tener un entorno local que replica Firebase Realtime
 Database y Firebase Authentication (Email-link passwordless)
 sin tocar los datos reales que usa tu familia ni la nube prod.

-------------------------------------------------------------
 0) PRE-REQUISITOS (unico setup manual)
-------------------------------------------------------------
   a) Node v22+ (ya lo tienes).
   b) firebase.json debe tener el bloque "emulators" con
      auth en :9099, database en :9000 y ui en :4000.
      Sin este bloque firebase-tools arranca solo RTDB y avisa:
        "!  auth: Not starting the auth emulator, make sure
              you have run firebase init."
      Puedes o bien:
        - Editar firebase.json a mano (recomendado; ya esta
          hecho en el commit chore(dev) que anadio este archivo).
        - O correr `npx firebase init emulators` y marcar
          auth + database + ui.
   c) En app.js la rama `?env=emul` ya conecta AMBOS emuladores:
        connectDatabaseEmulator(db,   'localhost', 9000)
        connectAuthEmulator   (auth, 'localhost', 9099)
      No hace falta tocar nada en el codigo.

-------------------------------------------------------------
 1) ARRANCAR EL EMULADOR (UNA SOLA LINEA)
-------------------------------------------------------------
   Desde la raiz del proyecto:

     npx firebase-tools emulators:start --only auth,database --project shopmate-e9195 --import=./emulator-data --export-on-exit

   Notas:
   - Una sola linea. El caracter \ sigue valiendo para
     continuacion en bash/zsh (incluye Git Bash en Windows),
     pero NO funciona en PowerShell o cmd.exe: lo interpretan
     como operador. PowerShell usa back-tick ` para continuar.
   - Sobre --import=./emulator-data: la primera vez la carpeta
     no existe (firebase-tools la acepta vacia, simplemente no
     importa nada). Para prepoblarla:
       Windows nativo:  mkdir .\emulator-data
       bash:            mkdir -p ./emulator-data
   - Sobre --export-on-exit: al parar (Ctrl+C) exporta
     ./emulator-data/database_export y ./emulator-data/
     authentication_export. La siguiente vez --import los
     restaura automaticamente.
   - El UI queda en http://127.0.0.1:4000 y ahi puedes:
       · ver items de /items/ en la pestana Realtime Database
       · ver usuarios Auth registrados + pending email-link
       · editar/borrar datos a mano para tests
   - Si en Windows el PATH no encuentra firebase-tools, asegurate
     de tener internet y `npx` resolvera el paquete on-the-fly.

   Salida esperada (resumen, sin warnings de "auth not starting"):

     i  ui:       Starting Emulator Suite UI on 127.0.0.1:4000
     i  auth:     Starting the Auth Emulator
     i  database: Starting the Database Emulator
     i  database: Importing data from .../emulator-data/database_export/...
     ┌──────────┬────────────────┬────────────────────────────┐
     │ Auth     │ 127.0.0.1:9099 │ http://127.0.0.1:4000/auth │
     │ Database │ 127.0.0.1:9000 │ http://127.0.0.1:4000/database │
     └──────────┴────────────────┴────────────────────────────┘

   Troubleshooting puerto ocupado:
   Si ves "address already in use" / EADDRINUSE en algun puerto
   (9099, 9000, 4000 o 4400), hay un emulador o servicio anterior
   ocupando ese puerto. Para liberarlo:
     - Linux/macOS:  lsof -i :9000  ->  kill <pid>
     - Windows bash: netstat -ano | findstr :9000  ->  taskkill /F /PID <pid>
     - Mas simple: Ctrl+C en la terminal vieja si la tienes abierta.

   Warning ignorable de CLI no autenticada:
   Si ves "You are not currently authenticated so some features may
   not work correctly. Please run firebase login": IGNORALO.
   El emulador local arranca sin necesidad de login OAuth (ese
   solo es obligatorio para `firebase deploy` a la nube).

-------------------------------------------------------------
 2) USARLO CON LA APP
-------------------------------------------------------------
   En otra terminal, sirve la app:

     python -m http.server 8080

   Y abre en el navegador la version con flag:

     http://localhost:8080/?env=emul

   - Sin ese flag, conecta a Firebase prod (nube familiar).
   - En consola del navegador (DevTools) veras:
       [firebase] usando EMULADOR local (RTDB:9000 + Auth:9099)
       [auth] instancia inicializada
   - El modal "Identificate" sale igual que en prod.
   - Cualquier operacion contra RTDB o Auth va contra el
     emulador local, sin tocar la nube.

-------------------------------------------------------------
 3) PROBAR AUTH (email-link) EN LOCAL
-------------------------------------------------------------
   El emulador Auth NO envia emails reales. Para testear el
   flujo end-to-end paso a paso:

   3.1 En la app, escribe tu email y pulsa "Enviar enlace".
       El modal cambia a "Revisa tu correo".
   3.2 Abre el UI del emulador: http://127.0.0.1:4000/auth
       En la pestana "Providers" veras la peticion pendiente.
   3.3 En los logs de la terminal donde arrancaste el
       emulador, busca una linea tipo:
         "To complete signin, open http://127.0.0.1:8080/
          ?apiKey=...&oobCode=...&mode=signIn"
       Copia esa URL y pegala en el navegador.
   3.4 La app detecta el link (isSignInWithEmailLink), valida
       con signInWithEmailLink y entra con sesion iniciada.
       [auth] signed-in uid=... email=...

   Si reinicias el emulador SIN --import, los usuarios
   pendientes y confirmados se pierden (es un emulador). Para
   persistir entre sesiones, manten --import + --export-on-exit.

-------------------------------------------------------------
 4) PARA Y LIMPIA
-------------------------------------------------------------
   Ctrl+C en la terminal del emulador. --export-on-exit
   persiste los datos automaticamente. Para borrar y empezar
   desde cero:

     rm -rf emulator-data/database_export emulator-data/authentication_export

   Y vuelve a arrancar el emulador.

-------------------------------------------------------------
 5) LIMITACIONES CONOCIDAS DEL EMULADOR
-------------------------------------------------------------
   - Email-link real no funciona: hay que copiar el link
     manualmente desde el UI o desde los logs del emulador.
   - El emulador RTDB evalua las mismas reglas que prod, pero
     `newData.numChildren()` puede rechazarse en v4.11+. Por
     eso MAX_ITEMS vive en app.js (defensa-en-profundidad),
     no en rules. Si te topas con errores de este tipo, el
     cap ya esta aplicado en el cliente.
   - No testees password reset / OAuth providers aqui: el
     emulador no los implementa sin configuracion extra.
   - Las reglas endurecidas con `auth != null` aplican tambien
     en el emulador. Para testear el flow end-to-end, primero
     completa un sign-in via email-link (paso 3).

   Debug cuando algo no va:
   Si Auth o RTDB se comportan raro en local, los logs estan en
   la raiz del proyecto:
     - firebase-debug.log   Arranque general + warnings de CLI
     - database-debug.log   Operaciones RTDB + reglas evaluadas
     - auth-debug.log       Sign-in, sign-out, link generado
     - ui-debug.log         UI del emulador
   Abre el que toque y busca lineas que empiezen por '!' (warning)
   o 'Error'.

-------------------------------------------------------------
 RESUMEN DE PUERTOS
-------------------------------------------------------------
   UI Emulator Suite:   127.0.0.1:4000  (UI web de todo)
   Auth Emulator:       127.0.0.1:9099  (sign-in por email-link)
   Database Emulator:   127.0.0.1:9000  (RTDB shopmate-e9195)
   Emulator Hub:        127.0.0.1:4400  (gestion interna)
